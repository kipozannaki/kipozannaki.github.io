一、引言：从工具到员工的关键一跃

工具需要人按开关。员工自己会看时间表、自己检查质量、干不好知道找谁报告。

前面六篇讲的是"怎么让 AI 干好一次任务"——给它手脚（Agent）、给它专业知识（Skill）、给它安全带（Harness Engineering）、给它干净的工作台（Context Engineering）、给它清楚的说明书（Prompt Engineering）。这些都是单次任务内的优化，还是得有人来按"开始"按钮。

Loop Engineering 要做的完全不同：设计一个持续运行、按计划触发、自我验证、知道什么时候该停、什么时候该交给人的调度系统。

具体到工作场景：你不是"有问题时手动启动代码审查"，而是"设置一条规则：每 6 小时自动检查一次代码库，发现新提交就启动审查流程，审查完验证结果，通过则记录成功、失败则自动重试一定次数、真搞不定就创建 Issue 通知人工"——整个过程不需要人从中间插手，系统自己转起来，一直转到该停的时候。

💡 Loop Engineering 的终极问题不是"怎么让它跑"，而是"怎么让它知道什么时候该停"。

二、Loop Engineering 在工程体系里的坐标

先用"什么时候需要 Loop"来理解它的位置：

　 手动触发 自动化 Workflow Loop 调度系统
启动方式 人点按钮 外部事件触发（如 webhook） 定时 + 事件 + 条件，自主判断何时启动
一次运行后 人看结果、人决定下一步 返回结果给上游系统 自己验证、自己决策、自己重试或交人
适用场景 一次性任务、探索 响应式处理、单次事件链 持续监控、周期性任务、长期自主运行

在本站六篇框架里的位置：

  • × Agent 到 Prompt Engineering　这五篇讲的都是"单个 Workflow 内部怎么优化"。Loop 不关心"这个 Workflow 能不能跑好"，只关心"这个 Workflow 跑完了怎么办、要不要再跑一次、什么时候能停"。
  • × Harness Engineering　下一节会单独展开。简单说，Harness 管"一个 Agent 运行的环境边界"，Loop 管"多个 Agent 运行的调度和结果仲裁"——两者是垂直分层的。

三、拆开一个 Loop，里面装的是什么

一个完整的 Loop 由五块不可分割的组件组成。少一块，整个系统就会在某个环节失控：

3.1 触发器（Trigger）：什么时候启动？

定时触发（Cron）：如"每 6 小时检查一次代码库"，适合周期性任务。

事件触发（Webhook）：如"GitHub 有新提交就启动审查"，适合响应式任务。

条件触发：如"上次运行发现 3 个未修复问题，2 小时后自动复查"，这是 Loop 层独有的——基于上次运行的结果来决定何时再跑一次。

关键决策：触发器本身可能失效（cron 没触发、webhook 丢失），所以 Loop 需要一个兜底检查："上次运行是什么时候？距离现在多久了？如果超过预期的最长间隔，说明触发器出问题了，需要主动告警。"

3.2 执行器（Executor）：怎么跑？

负责启动 Workflow 的实例。关键决策包括：

  • 串行还是并行？同一个 Loop 的两个实例能否同时跑？（通常答案是"不能"，除非 Workflow 之间完全无状态共享）
  • 资源隔离：每个 Workflow 实例是否要独立的 Harness 环境、独立的工具连接？
  • 超时控制：单个 Workflow 实例最长能跑多久？超时了是强行杀掉，还是让它跑完再判断？

3.3 验证器（Validator）：结果行不行？

这是最容易被忽视、也最容易出错的环节。关键原则：验证必须独立于执行，不让 Workflow 自己判断自己（这跟 Harness Engineering 里"生成者-评估者分离"是同一个思路）。

常见的验证方式：

  • 规则验证：输出是否符合预定义的 schema？（如代码审查输出是否包含"问题清单"和"修复建议"两个字段）
  • 测试验证：跑一套测试套件，是否通过？
  • 一致性验证：同样的输入，多次运行 Workflow 的输出是否一致？如果不一致说明系统有随机性或状态问题。
  • 人工验证：抽样检查，人工标注"通过/不通过"——这个费人力，但对关键任务最可靠。
  • 模型验证：用另一个模型评估输出质量（代价高，慎用）。

3.4 决策器（Decider）：下一步干啥？

根据验证结果，决定 Loop 的下一步动作。关键：决策逻辑必须可配置、可观测——绝对不能写死在代码里。

标准的决策矩阵：

验证结果 决策类型 动作
通过 成功终止 记录成功，停。下次触发时跳过已处理的部分。
不通过，可重试 重试 修改触发条件或补充信息，再启动一次 Workflow
不通过，不可重试 降级 降低目标或缩小范围，再试一次
连续 N 次不通过 交人 终止 Loop，创建 Issue 或通知人工介入

3.5 状态持久化（State）：记忆什么？

Loop 需要跨多次运行的记忆，包括：

  • 运行历史：每次 Workflow 的输入、输出、验证结果、耗时、成本
  • 重试计数：当前已重试几次，上限是多少
  • 问题模式库：历次运行发现的共性问题，用于下次优化触发条件
  • 性能指标：平均运行时长、成功率、Token 消耗趋势
  • 处理进度：比如代码审查 Loop，各文件的审查状态是"已审""待审"还是"需重审"

四、真实案例：一个代码库的持续审查 Loop

把五大组件组合在一起，看看一个真实的 Loop 系统怎么运转。这个案例改编自我的开源项目 RELOO，但展示的是"如果 RELOO 被嵌入到一个 Loop 调度系统里"会长什么样。

案例：GitLab CI 集成的代码审查自动化

业务需求：一个代码库，每当有新提交时，需要自动运行多轮代码审查。如果审查发现问题，自动尝试修复。修复后再次审查，确保质量达标。

流程图：

[Loop Trigger]
  每 6 小时检查一次 GitLab
  或 GitLab webhook 触发（新 commit/PR）
           │
           ▼
[Executor]
  启动 "RELOO 审查 Workflow" 实例
  隔离的 Harness 环境（独立沙箱）
  超时上限：30 分钟
           │
           ▼
[Workflow 内部] ← 这是 RELOO 的地盘
  └─ 六轮审查（核心逻辑、状态管理、交互一致性……）
  └─ 每轮有子级的验证和重试
           │
           ▼
[Validator]
  1. Schema 验证：审查输出包含"问题清单"吗？
  2. 一致性验证：相同代码重跑一次，输出一致吗？
  3. 测试验证：修复后的代码能跑测试吗？
           │
           ├─ 全通过 ──→ [Decider] 成功终止
           │              └─ 记录"第 5 次运行成功"
           │              └─ 标记这些文件"已审查"
           │              └─ 下次触发时跳过
           │
           └─ 失败 ──→ [Decider] 判断重试还是交人
                      │
                      ├─ 重试计数 < 2 ──→ [Executor] 再跑一次
                      │                    （这次只聚焦失败维度）
                      │
                      └─ 重试 ≥ 2 ──→ 创建 GitLab Issue
                                      "自动审查失败，请人工审查"

关键洞察：

  • Loop 层不决定"代码审查怎么做"（那是 Workflow/Harness 的事），只决定"什么时候开始审查、审查完了谁来验、验完了怎么办"
  • RELOO 的"六轮审查"是 Workflow 内部的细节，Loop 只关心"输入"和"输出是否达标"
  • 一旦任务规模从"一次性审查一个文件"变成"持续监控整个代码库"，Loop 层的价值才真正体现出来

五、实战避坑：两个"血泪翻车现场"

坑一：没有终止条件，Loop 变成了"永动机"

翻车现场还原：

某个 Loop"验证通过就继续跑，不通过就重试"，但没有明确的"最多重试几次"上限。结果一个 Workflow 实例卡在了某个边界情况（比如网络抖动导致一个外部 API 频繁返回错误），决策器一直选择"重试"，Loop 就这样转啊转，烧钱不停。

根因剖析：

没有硬性的终止条件。没有"连续 N 次失败就交人、没有重试次数上限、没有总运行时间上限"。

解决方案：

在决策器里硬编码四种终止条件，缺一不可：

终止类型 条件 示例
成功终止 连续 N 次 Workflow 输出通过验证 "连续 3 次代码审查未发现新问题"
失败终止 重试次数达到上限 "已重试 3 次，仍不通过，交人工"
超时终止 总运行时间超过阈值 "这个 Loop 已经跑了 24 小时，强制停"
资源终止 成本/Token 消耗超限 "本月审查预算已用完，停"
💡 没有终止条件的 Loop 不是自动化，是自动化灾难。

坑二："Loop 级"失败被当成"Workflow 级"失败处理

翻车现场还原：

触发器失效了（cron 没触发、webhook 丢失），但系统没有发现。几天之后，某人才意外发现代码库的新提交压根没被审查过。

根因剖析：

系统只在乎"单次 Workflow 有没有异常"，没有在乎"这个 Loop 本身有没有按时跑"。这是一个更高层级的失败检测。

解决方案：

在 Loop 层单独加一套"健康检查"逻辑：

def loop_health_check(loop_state):
    """检测 Loop 层本身有没有出问题"""
    last_run = loop_state['last_trigger_time']
    expected_interval = loop_state['trigger_interval']  # 比如 6 小时

    if (now - last_run) > expected_interval * 1.5:
        # 没有按时触发，可能是触发器出问题了
        alert("Loop health check failed: No run in the past {interval}")
        # 手动执行一次备用流程
        emergency_execute_workflow(loop_state)

类似的高层级失败还包括：

  • 验证器失效：验证逻辑本身有 bug。解决方案：用对照组（两套不同的验证器，结果要一致）
  • 决策器死循环：决策逻辑进入"总是决定重试"的死循环。解决方案：任何情况都必须有"最后的杀招"——实在不行就交人
  • 状态损坏：持久化状态丢失或不一致。解决方案：定期快照 + 异常时的恢复机制
💡 Loop 级失败往往比 Workflow 级失败更隐蔽、危害更大，因为系统表面看起来还在"跑"，其实已经瞎跑了。

六、进阶：Loop / Workflow / Harness 的层级关系

这三个概念经常被搞混，因为它们确实有重叠。但用一张表来看，各自职责很清楚：

　 Harness Engineering Workflow Loop Engineering
时间维度 单次 Agent 运行内 单次运行的多步骤编排 跨多次运行的调度和决策
核心问题 Agent 怎么安全地自主行动？ 多个步骤怎么连成一条链？ 多次运行怎么协调、什么时候停？
关键组件 权限边界、验证回路、重试机制 任务定义、步骤顺序、数据流 触发器、验证器、决策器、状态管理
失败处理 单步骤出错怎么办 某个 Workflow 实例失败怎么办 多次都失败了怎么办、什么时候认输
例子 给 Agent 加计数器防死循环 设计"查数据 → 处理 → 生成报告"的三步链 每天自动跑这个 Workflow，失败自动重试，5 天还失败就告诉人

环境隔离与资源管理（这是 Loop 层和 Harness 的交界处）：

  • Loop 启动每一个 Workflow 实例时，都需要给它分配一个独立的 Harness 环境（独立沙箱、独立工具连接）吗？
    • 不一定——如果 Workflow 之间完全无状态共享，可以复用同一个 Harness
    • 但通常要隔离——防止一个 Workflow 实例的异常行为污染另一个
  • Loop 终止后，分配给它的 Harness 环境怎么清理？
    • 必须显式回收——否则资源泄漏，时间长了系统会窒息

七、结语：八块拼图拼完

给小白的话：Workflow 让 AI 能连续干一串事情（第 1 步查数据、第 2 步分析、第 3 步写报告）。Loop 让 AI 能一直干下去——不用你每次手动点"开始"。它像一个勤劳的实习生：每天按时来上班、自己检查工作质量、干得好就继续、干不好就重做、实在搞不定就告诉你"老板，这个我真不会"。

给开发者的话：Loop 层是 AI 系统从"工具"变成"员工"的关键一跃。工具需要人时刻在旁边按开关。员工自己会看时间表、自己报告进度、知道什么时候该停。但代价是，你设计 Loop 时最难的部分，不是"怎么让它跑起来"，而是"怎么让它知道什么时候该停、什么时候出错了要告诉人"——这是系统可靠性和成本的分水岭。

系列总结：回顾这六篇（加上即将推出的底层基础专题），整套 AI 工程的坐标系其实很清晰：

层级 涉及的概念 解决什么问题
底层基础 Transformer / 预训练 / Tokenization / RLHF 模型本身怎么造出来的
单次优化层 Prompt / Context / Harness / Agent / Skill 让一次任务做好
持续运行层 Loop Engineering 让任务一直做下去、知道什么时候停
💡 没有 Loop，AI 系统始终是工具。有了 Loop，它才开始有"自主意识"——知道自己该干什么、干得好不好、什么时候该向人类交接。
