一、引言：从工具到员工的关键一跃

工具需要人按开关。员工自己会看时间表、自己检查质量、干不好知道找谁报告。

前面六篇讲的是"怎么让 AI 干好一次任务"——给它手脚（Agent）、给它专业知识（Skill）、给它安全带（Harness Engineering）、给它干净的工作台（Context Engineering）、给它清楚的说明书（Prompt Engineering）。这些都是单次任务内的优化，还是得有人来按"开始"按钮。

Loop Engineering 要做的完全不同：设计一个持续运行、按计划触发、自我验证、知道什么时候该停、什么时候该交给人的调度系统。

💡 Loop Engineering 的终极问题不是"怎么让它跑"，而是"怎么让它知道什么时候该停"。

二、Loop Engineering 在工程体系里的坐标

手动触发自动化 WorkflowLoop 调度系统
启动方式人点按钮外部事件触发定时 + 事件 + 条件，自主判断何时启动
一次运行后人看结果、人决定下一步返回结果给上游系统自己验证、自己决策、自己重试或交人
适用场景一次性任务、探索响应式处理持续监控、周期性任务、长期自主运行

三、拆开一个 Loop：五大组件

3.1 触发器（Trigger）

定时触发（Cron）、事件触发（Webhook）、条件触发（基于上次运行结果）。兜底检查："上次运行是什么时候？超过预期间隔则告警。"

3.2 执行器（Executor）

负责启动 Workflow 实例：串行还是并行？资源隔离？超时控制？

3.3 验证器（Validator）

验证必须独立于执行，不让 Workflow 自己判断自己。规则验证 / 测试验证 / 一致性验证 / 人工验证 / 模型验证。

3.4 决策器（Decider）

验证结果决策类型动作
通过成功终止记录成功，停
不通过，可重试重试修改条件再启动
不通过，不可重试降级缩小范围再试
连续 N 次不通过交人创建 Issue 通知人工

3.5 状态持久化（State）

运行历史、重试计数、问题模式库、性能指标、处理进度。

四、真实案例：代码库的持续审查 Loop

[Loop Trigger]
  每 6 小时检查一次 GitLab
  或 GitLab webhook 触发
           │
           ▼
[Executor] 启动审查 Workflow 实例
           │
           ▼
[Validator] Schema 验证 + 一致性验证 + 测试验证
           │
           ├─ 全通过 ──→ [Decider] 成功终止
           └─ 失败 ──→ [Decider] 重试或交人

五、实战避坑：两个"血泪翻车现场"

坑一：没有终止条件，Loop 变成了"永动机"

四种终止条件缺一不可：成功终止 / 失败终止 / 超时终止 / 资源终止。

💡 没有终止条件的 Loop 不是自动化，是自动化灾难。

坑二："Loop 级"失败被当成"Workflow 级"失败处理

触发器失效、验证器有 bug、决策器死循环、状态损坏——这些是 Loop 级的问题，需要单独的"健康检查"逻辑。

六、Loop / Workflow / Harness 的层级关系

HarnessWorkflowLoop
时间维度单次 Agent 运行内单次运行的多步骤编排跨多次运行的调度和决策
核心问题Agent 怎么安全地自主行动？多个步骤怎么连成一条链？多次运行怎么协调、什么时候停？
失败处理单步骤出错怎么办某个实例失败怎么办多次都失败了怎么办

七、结语：八块拼图拼完

给小白的话：Workflow 让 AI 能连续干一串事情。Loop 让 AI 能一直干下去——不用你每次手动点"开始"。

给开发者的话：Loop 层是 AI 系统从"工具"变成"员工"的关键一跃。最难的部分不是"怎么让它跑起来"，而是"怎么让它知道什么时候该停"。

层级涉及的概念解决什么问题
底层基础Transformer / 预训练 / Tokenization / RLHF模型本身怎么造出来的
单次优化层Prompt / Context / Harness / Agent / Skill让一次任务做好
持续运行层Loop Engineering让任务一直做下去、知道什么时候停
💡 没有 Loop，AI 系统始终是工具。有了 Loop，它才开始有"自主意识"。
