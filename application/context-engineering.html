一、引言：为什么"塞得越多"反而越蠢

很多人对上下文窗口有个朴素的误解：反正窗口够大，多塞点信息总没坏处，让模型自己挑重点就是了。事实恰恰相反——学界给这个现象起了个名字，叫"Lost in the Middle"：把一份很长的资料塞进上下文，模型对开头和结尾的内容处理得明显更好，埋在中间的信息，哪怕完全相关，也容易被直接漏看。还有研究把这个现象进一步归纳成"Context Rot"（上下文腐化）——输入的 token 越长，模型的表现越会打折扣，不是因为信息不相关，纯粹是"看"得越多、"看"得越粗。

这个坑我们在 agent.html 里已经真刀真枪踩过一次了：工具返回的原始数据不加裁剪地全塞进上下文，一次请求的 token 量和延迟直接指数级暴涨。Context Engineering 要解决的问题比那次踩坑更进一步——不只是"别塞太多"，而是要系统性地回答：推理这一刻，模型的工作台上到底该摆哪些东西？这也是 Anthropic 自己给这门学问下的定义式问题：什么样的上下文配置，最可能让模型产出你想要的行为。

💡 上下文不是仓库，是工作台——这句话我们在 agent.html 里说过一次，这篇文章讲的就是怎么把这句话落成一套方法。

二、Context Engineering 在工程体系里的坐标

先分清楚 Context Engineering 和 Prompt Engineering 差在哪——这是最容易被混为一谈的一对概念：

　 Prompt Engineering Context Engineering
管的是什么 这句话该怎么写 这一刻该不该让模型看到这句话
作用时机 写提示词的时候，静态、一次性 贯穿整个推理过程，动态、持续
典型手艺 措辞、示例、角色设定 压缩、检索、隔离、加载策略
类比 写好一份说明书 决定这一刻工作台上摆哪几样工具

放进本站的坐标系里，Context Engineering 和另外五个概念是这样咬合的：

  • × Agent　Agent 跑的循环越长，越容易被自己制造的"信息噪音"淹没——每一轮工具调用的结果都在往上下文里堆，不管好坏都由 Context Engineering 负责筛。
  • × Skill　skill.html 讲的渐进式披露——先只加载 name 和 description，命中了才读正文——本质就是 Context Engineering 在"要不要现在读这份资料"这个决策上的具体打法。工具加载也是同一个原则：常用的几个工具常驻上下文，长尾工具靠动态发现按需拉入，和 Skill 的三层加载是同一套思路的两种实现。
  • × Prompt Engineering　见上面的对比表——一个管"写什么"，一个管"这一刻该不该给"。两者经常被当成一回事，其实是两种完全不同的手艺。
  • × Harness Engineering　harness-engineering.html 里说过，Context Engineering 是 Harness 里负责"喂饭"的子系统——什么时候压缩、什么时候整体重置，是 Context Engineering 的具体决策，但决策的触发规则要由 Harness 层来定。
  • × Loop Engineering　上下文该留多少、该在哪个节点摘要，没有放之四海皆准的公式，只能靠"改一版策略、跑一轮、看指标"逐步逼近——这仍然是 Loop Engineering 的活。

四、实战避坑：两个"血泪翻车现场"

坑一：关键规则埋在中间，模型直接"看不见"

翻车现场还原：

把一份几百行的规范文档一股脑塞进系统提示词，其中第 180 行写着一条硬性规则——"涉及金额的字段必须保留两位小数"。结果模型十次有六次就是没遵守。反复检查提示词，那条规则明明白白写在那儿。

根因剖析：

不是模型没看到文字，是模型对超长上下文里靠前、靠后的部分处理得明显更好，中间部分容易被"糊过去"——这就是前面提到的 Lost in the Middle。信息完全相关，位置不对，照样会被漏掉。

解决方案：

硬性规则、必须遵守的约束，放在上下文的头部或尾部，别埋在长文档的正中间；超长参考资料别整段糊进去，拆成按需检索的小块，需要哪条就拉哪条，而不是一次性摊开在模型面前指望它自己挑。

💡 越是重要的规则，越不能靠"塞在文档里"来指望模型自己发现——你得把它摆在模型一定会看的位置。

坑二：工具"货架"堆太满，模型选不准

翻车现场还原：

一个 Agent 挂了几十个工具，每个工具的 Schema 描述都占不少 token。模型经常调错工具，或者干脆在几个相似工具之间反复横跳——不是模型笨，是候选项太多，判断负担太重。

根因剖析：

每加一个工具的 Schema，都是实实占用上下文预算的成本；工具越多，模型要在候选项之间做区分的负担越重，选择准确率反而下降。

解决方案：

参考"常驻 + 按需发现"的分层——最常用的 3~5 个工具常驻上下文，其余的通过动态检索机制在真正需要时才拉进来。这和 Skill 的三层加载是同一个原则的两种实现：都是先给一个薄薄的目录，命中了才展开细节。

💡 工具不是越多越强大，货架堆太满，顾客反而找不到东西。

五、进阶：一个简单的"上下文预算"心法

每次想往上下文里加一样东西——一段文档、一个工具、一轮历史对话——先问自己一句：模型这一步真的需要看到它吗？而不是本能地觉得"信息多给点总没错"。

这句话听着简单，但和大多数人的直觉是反着来的。多数人对上下文的态度是"覆盖面优先"——生怕漏了什么，宁可多塞。但前面两个坑已经说明了：塞得全，不等于塞得对。上下文预算和金钱预算是一个道理——不是有多少花多少，是先想清楚这一步到底要买什么。

六、结语：上下文管理才是 Agent 的隐藏难度

给小白的话：下次发现某个 AI 产品"聊着聊着开始变傻"，大概率不是模型突然变笨了，是它的工作台被堆得太满，重要的东西被埋没了。

给开发者的话：别把上下文窗口当成"越大越安全"的免费仓库。写 Agent 之前先问自己：这一步，模型的工作台上真正需要哪几样东西？答不上来，说明这个 Agent 的上下文管理还停留在"能跑就行"的阶段。

💡 好的上下文工程，不是让模型看到更多，而是让它在需要的那一刻，正好看到该看的东西。
