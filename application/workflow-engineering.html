一、引言：为什么 Workflow 不是"把几个步骤串起来"

很多人对 Workflow 的理解停留在一张简单的流程图：步骤 A → 步骤 B → 步骤 C → 完成。结果上线后发现：步骤 B 的输出格式步骤 C 接不住、中间某步失败了不知道回退到哪、整个流程跑完了但结果根本不对。

问题出在哪？这三个地方都可能：

  • Plan 没想清——步骤拆得太粗，一个步骤里塞了太多事；或者拆得太碎，步骤之间来回传数据，耦合得像 spaghetti
  • Spec 没写明白——"步骤 B 处理数据"，处理成什么样算对？输出格式是什么？失败怎么办？没定义清楚，等于没定义
  • 状态没管好——步骤 A 改了全局变量，步骤 B 读了以为是对的，其实步骤 A 中途失败了没回滚，数据已经脏了

Workflow Engineering 干的事：从"怎么拆步骤"开始、经过"每一步的验收标准是什么"、"步骤之间怎么传数据"、"错了怎么回退"，设计一套从输入到输出的可验收协作系统。Workflow 不是一张流程图，而是一套有 Plan、有 Spec、有状态管理、有错误处理的工程规范。

💡 Workflow 是"让多个步骤协作完成一件事"，但怎么拆、每一步做到什么程度算对、错了怎么修，每一步都需要设计。

二、Plan：步骤序列的"拓扑图"

Plan 回答的是：这件事需要分几步？哪些步骤可以并行？步骤之间的依赖关系是什么？

Plan 不是"怎么做"，而是"有哪些步骤、什么顺序、什么依赖关系"。

2.1 拆步骤的原则

原则一：一个步骤只做一件事

反例："审查代码并生成修复建议并提交 PR"——这是三件事。拆成三个步骤：审查 → 生成建议 → 提交 PR。

原则二：步骤的边界由数据流决定

什么时候该拆？看数据：如果步骤 A 的输出是步骤 B 的输入，且中间数据需要被验证，那就该拆。

原则三：并行还是串行，看依赖不看直觉

直觉上"先查数据再分析"是串行，但如果"查数据"可以拆成"查数据库"和"查缓存"两个独立操作，它们就可以并行。Plan 的关键是画出依赖图（DAG），而不是时间线。

2.2 Plan 的表示方式

Workflow: 代码审查自动化
├── 步骤 1: 获取提交差异 (输入: commit SHA, 输出: diff)
│   └── 依赖: 无
├── 步骤 2: 静态分析 (输入: diff, 输出: 分析报告)
│   └── 依赖: 步骤 1
├── 步骤 3: 安全扫描 (输入: diff, 输出: 安全报告)
│   └── 依赖: 步骤 1  (与步骤 2 并行)
├── 步骤 4: 汇总结果 (输入: 分析报告 + 安全报告, 输出: 审查报告)
│   └── 依赖: 步骤 2, 步骤 3
└── 步骤 5: 提交评论 (输入: 审查报告, 输出: PR 评论)
    └── 依赖: 步骤 4

三、Spec：每个步骤的"验收标准"

Spec 回答的是：这一步的输入是什么、输出是什么、算对了的标准是什么、错了怎么办。

3.1 整体 Workflow 的 Spec

【Workflow 整体 Spec】
输入: GitHub 提交事件（commit SHA）
输出: 审查报告（markdown）+ 是否允许合并（boolean）

【整体验收标准】
- 所有 3 个检查维度都必须完成（不能因为某个失败就跳过其他）
- 报告必须在 10 分钟内生成
- 如果有阻塞性问题，必须有"后续步骤"的建议

【整体失败处理】
- 某个检查超时 → 标记为"检查未完成"，不阻塞其他检查
- 汇总阶段失败 → 回退到前一个 checkpoint，重新汇总
- 整体超时 → 生成部分报告，标记为"超时中止"

3.2 单个步骤的 Spec 模板

步骤 4: 汇总结果

【输入】
- 分析报告（JSON，含问题列表和严重程度）
- 安全报告（JSON，含漏洞列表和风险等级）

【输出】
- 审查报告（Markdown，含摘要、详细问题、修复建议）
- 是否阻塞合并（boolean）

【验收标准】
- 报告必须包含"摘要"、"详细问题"、"修复建议"三个部分
- 阻塞性问题的判定标准：严重程度为"critical"或安全等级为"high"
- 报告长度不超过 2000 字

【失败处理】
- 输入格式不对 → 尝试解析，解析失败则标记为"输入异常"
- 生成报告超时（> 2 分钟）→ 输出简化版报告，标记"部分生成"
- 输出不符合 schema → 重新生成一次，仍失败则人工介入

关键：Spec 必须可验证。"输出一份好报告"不是 Spec，"报告必须包含三个部分、不超过 2000 字"才是。

四、Workflow 在工程体系里的坐标

层级管什么与 Workflow 的关系
Prompt单轮输入怎么写Workflow 的每个步骤都需要 Prompt 来告诉模型干什么
Context单次会话里喂什么Workflow 的状态可能很大，Context 负责只喂必要信息
RAG外部知识怎么接进来Workflow 步骤可能需要调用 RAG
Agent单个自主实体Workflow 的每个步骤通常是一个 Agent 或 Agent 的一次调用
Skill打包可复用能力Workflow 可以把整个 Workflow 本身打包成一个 Skill
Harness单次 Agent 运行的环境Workflow 的每个步骤运行在 Harness 中
Loop跨多次运行的调度Loop 触发/管理 Workflow 实例

五、设计模式：不只是"步骤 A → 步骤 B"

不同场景应该选什么模式？用这个决策树：

Workflow 需要在运行时根据结果动态改变路径？
├─ 是 → 状态机（允许分支、条件跳转）
└─ 否 ↓

步骤之间有依赖？能否并行执行？
├─ 完全线性、无法并行 → 线性链
├─ 有部分分支可并行 → DAG
└─ 完全独立、解耦为上 → 事件驱动

六、状态管理：Workflow 的"记忆"

Checkpoint 机制：在每个关键步骤完成后，保存当前状态，中断后从断点恢复。

七、错误处理：Workflow 的"安全气囊"

失败类型表现策略
瞬时失败API 超时、网络抖动指数退避重试，最多 3 次
逻辑失败输出格式错、不符合 schemafallback_chain：重试 → 换模型 → 人工
不可恢复失败权限不足、资源不存在立即终止，报告错误
依赖失败上游步骤失败关键依赖终止，可选依赖降级
Workflow 级失败状态损坏、超时、资源耗尽Checkpoint 恢复 / Cleanup / 人工

八、主流方案对比

框架核心特点适用场景缺点
LangGraph与 LangChain 生态深度集成，图结构定义 Workflow快速迭代、LLM 密集型 Workflow不适合需要企业级可靠性的场景
Temporal企业级持久化、分布式执行、自动重试和补偿长流程、关键业务、需要中断恢复学习曲线陡、基础设施重
LlamaIndex Workflows面向 RAG 场景，与 LlamaIndex 生态集成RAG 密集型、数据流复杂的 Workflow新生态，社区沉淀少
自研完全可控，可定制审计日志、特殊权限管理有强定制需求、已有成熟基础设施维护成本高

十二、结语：Workflow 是"协作"的基础

给小白的话：Workflow 就是"让多个步骤协作完成一件事"。但不是简单地把步骤串起来，而是要想清楚：这件事分几步、每步做什么、做到什么程度算对、错了怎么办。

给开发者的话：Workflow 系统的性能瓶颈很少在"步骤执行有多快"，而在"步骤之间的数据流是否清晰、Spec 是否可验证、错了是否能恢复"。优化 Workflow 要从设计阶段开始——Plan 画清楚依赖图、Spec 写明白验收标准、状态管理选好持久化策略、错误处理分好层。

💡 Workflow 不是万能药，但它是现代 AI 应用从"玩具"变成"工具"的关键。
