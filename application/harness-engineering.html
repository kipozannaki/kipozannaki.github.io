从"祈祷提示词"到"工程化兜底" · 双栖科普

一、引言：为什么"裸奔"的 Agent 会要命

特种兵光有枪和目标还不够。没有装备手册、没有战地规则、没有指挥部随时喊停——照样会打偏、会走火、会把自己人炸了。这就是"裸奔"的 Agent：有模型、有工具，却没有 Harness。

不信邪？Anthropic 自己做过一组对照实验：同一个模型 Opus 4.5，同一句提示"做一个 2D 复古游戏编辑器"。不套任何脚手架、让模型单跑：20 分钟，花 9 美元，产出勉强能打开但基本没法真正玩。换成一套完整的 Harness——拆成规划者、生成者、评估者三个角色分工协作：跑了 6 个小时，花了 200 美元，最后交出来的是一个功能完整、真能玩的产品。同一个大脑，穿没穿"盔甲"，产出是天差地别的两个物种。

这组数字戳破了一个常见误解：很多人觉得 Agent 效果不好是模型不够聪明，加把劲优化 Prompt 就行。但上面这个实验里，模型完全没变——变的是模型运行所在的那套系统。Harness Engineering 干的就是这件事：设计模型工作时所处的整个外部环境——上下文怎么喂、工具接口怎么设计、结果由谁验收、权限边界划在哪、出了错怎么兜底。

💡 模型的能力是天花板，Harness 决定的是这个天花板底下，你能不能稳定地摸到它。

二、Harness Engineering 在工程体系里的坐标

先把"裸跑"和"全副武装"之间的差距摆在台面上：

　 裸跑（无 Harness） 基础兜底 完整 Harness（规划 + 生成 + 评估）
执行方式 模型一次性输出 加了重试 / 超时 角色分工，独立会话互相制衡
谁来验收 没人，模型自己说"做完了" 简单规则校验 独立评估者，看不到生成过程
参考成本 / 耗时* 9 美元 / 20 分钟 — 200 美元 / 6 小时
适合什么任务 一次性草稿、低风险探索 中等重要、可人工兜底 要长期跑、错了代价大

*数据来自 Anthropic 官方实验：同一模型、同一个"2D 复古游戏编辑器"任务，仅 Harness 结构不同。

把这把尺子放进本站的坐标系里，Harness Engineering 和另外五个概念的关系是这样的：

  • × Agent　agent.html 里说过"Agent 越自主，Harness 就该收得越紧"，反过来同样成立：没有 Harness 兜底的 Agent，本质上只是一段运气好才能跑通的脚本——就是那个凌晨两点因为死循环烧了 200 块的翻车现场。
  • × Skill　Skill 是装进系统里的专业知识包，Harness 是决定这个知识包什么时候被读取、以什么权限被执行的运行时环境。SKILL.md 里能声明 allowed-tools，管的正是 Harness 这一层的事。
  • × Context Engineering　Harness 里最重的一块活儿就是上下文管理——什么时候压缩、什么时候整体重置、怎么让评估者拿到一个"干净"的上下文而不是被生成过程污染。可以说 Context Engineering 是 Harness 里专门负责"喂饭"的子系统。
  • × Prompt Engineering　同一套 Harness，换一代模型可能就要拆掉一半——下面第四节会讲到，靠 Prompt 撑住的复杂分段结构，换了更聪明的模型反而变成累赘。Prompt 负责让模型"听懂"，Harness 负责让整套系统"扛得住"。
  • × Loop Engineering　一次搭好的 Harness 大概率是错的——好的做法是一次只拆一个组件、跑一轮、看少了它任务还成不成。这个"改一个变量、测一轮、留下有用的"的过程，就是 Loop Engineering 在 Harness 场景下的样子。

三、拆开一副称手的 Harness，里面装的是什么

把"Harness"这个词拆开看，其实就是五块拼图：

组件 负责什么 缺了它会怎样
上下文管理 什么时候压缩、什么时候整体重置 上下文塞爆，模型开始"失忆"
工具接口设计 参数怎么定义、报错怎么表达 报错含糊，模型瞎重试
验证 / 评估回路 由谁、按什么标准判断"做完了" 模型自己给自己打满分
权限与沙箱 哪些操作允许、哪些要拦下 越权操作、误删数据
规划与状态追踪 怎么让下一个 session 接得上 每次从零开始，反复返工

"规划与状态追踪"这块最容易被低估。跨多轮 session 干活的 Agent，就像轮班的工程师——新来的这一班，对上一班干了什么一无所知。工业级做法是让第一轮的"初始化 Agent"把需求拆成几十上百条可验证的功能点，全部先标记"未完成"，后面每一轮的 Agent 就拿这份清单当交接班记录：

{
  "features": [
    { "id": "F001", "desc": "用户可以创建新关卡并保存", "status": "failing" },
    { "id": "F002", "desc": "点击播放按钮进入可玩模式", "status": "failing" },
    { "id": "F003", "desc": "关卡编辑器支持拖拽放置精灵", "status": "failing" }
  ]
}

不用靠"猜项目做到哪了"，直接看这份清单里还剩几条 failing。

四、实战避坑：两个"血泪翻车现场"

坑一：让模型自己判自己的作业，它是个"乐观诈骗犯"

翻车现场还原：

Agent 干完活，你问它"做完了吗"——十有八九斩钉截铁地回答"完成了！"。你打开一看，功能缺一半，甚至代码根本跑不起来。

根因剖析：

让模型评估自己的产出，就像让考生自己判卷——它对自己的工作天然是个盲目乐观的裁判，尤其在没有客观对错、只有质量高低的任务上（设计、文案、代码风格），这个偏差会更明显。

解决方案：生成者与评估者分离

把"干活"和"验收"拆成两个互不知情的角色：评估者拿到的是一个从没见过生成过程的干净上下文，只看最终产出，按明确标准打分——功能对不对、有没有漏掉的边界情况、风格是不是套路化。这就是 Anthropic 那套生成者-评估者架构的核心，思路上和 GAN 里生成器与判别器互相较劲很像。

💡 不要让运动员自己当裁判——哪怕这个运动员是全世界最强的模型。

坑二：Harness 会"过期"——旧模型的枷锁，套在新模型身上就是负担

翻车现场还原：

一套为"上一代模型"精心调校的复杂 Harness——分好几个阶段、频繁重置上下文、层层校验——换上更强的新模型之后，任务不仅没变快，反而因为多余的编排环节拖慢了进度，白烧 Token。

根因剖析：

Harness 里的每一个组件，本质上都是在为模型的某个弱点打补丁。模型这一代做不到的事，下一代可能轻松搞定——比如某个版本容易在长任务里"没做完就急着收尾"，那道靠频繁重置上下文续命的补丁，到了改掉这个毛病的新模型手里就是纯粹的累赘。

解决方案：换模型就做一次"拆解测试"

每次升级底层模型，别急着照抄旧 Harness，逐个组件拿掉，看少了它任务还完不完得成。能拿掉的果断拿掉——这是 Loop Engineering 那套"改一个变量、跑一轮、看结果"的方法论，用在 Harness 维护上。

💡 Harness 不是搭好就一劳永逸的——它得跟着模型一起"发育"，你今天的解药，可能是明天的镣铐。

五、进阶：怎么判断这个任务要不要上 Harness

不是所有任务都值得套完整的三角色 Harness——Anthropic 自己的设计原则是：找最简单能跑通的方案，只有真正需要时才增加复杂度。一个粗糙但够用的判断依据：

  • 任务是一次性的、错了代价小、人工能顺手兜底 → 裸跑或加个基础重试就够，别为了"显得专业"堆结构
  • 任务要跨很多轮 session、错了代价大、没人盯着结果 → 值得上完整 Harness，成本会比你想的更快赚回来

那组 9 美元 vs 200 美元的数字放在这里正合适：如果你要的是一次性草稿，9 美元的方案完全够用；如果你要的是能交付的产品，200 美元买的不是"更贵"，是"能用"和"不能用"之间的那道坎。

六、结语：Harness 才是 Agent 的真骨架

给小白的话：下次用一个 AI 产品觉得"这个 AI 好靠谱"，靠谱的多半不只是模型聪明，背后大概率有一整套你看不见的 Harness 在替它兜底。这个词听着唬人，说白了就是"给 AI 配的工作规范和安全带"。

给开发者的话：别只盯着 Prompt 反复调优——找一个你正在跑的 Agent，拆开看看它有哪一步是靠"祈祷模型别出错"硬撑着的，那一步就是你该补 Harness 的地方。上下文怎么管、谁来验收、权限边界在哪、状态怎么交接，这四个问题答不上来，说明这个 Agent 还是在"裸奔"。

💡 模型是骨架的血肉，Harness 才是让这具身体不散架的骨架本身。
