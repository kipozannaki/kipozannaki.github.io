🛠️ Skill

AI 工具能力的最小单元

三国故事里，谋士出征前常会留一个锦囊，上面写着"遇某种情形，方可拆看"。平时它安静地放在怀里，不占地方，只有情况对上了，才会被打开、照做。

Skill 差不多就是这个逻辑：AI 平时只记得"我怀里揣着哪些锦囊、分别是干什么用的"，真正遇到对应场景，才会把某一个锦囊完整展开，照着里面写的方法去做。

核心机制

一个 Skill 就是一个文件夹，最少只需要一个 SKILL.md 文件，也可以再加脚本、参考文档、模板资源。它不给模型增加新的智能，而是把"这件事该怎么做"的经验，从一次性的对话指令，变成可复用、可分享、可版本管理的资产。

它的关键设计是渐进式披露，分三层加载：

  1. 常驻层：只有 Skill 的名字和一句话描述，一直挂在系统提示里，几乎不占上下文
  2. 命中加载层：判断和当前任务相关后，才把 SKILL.md 正文读进来
  3. 按需深挖层：正文里提到的脚本、参考文档，只有真正用到时才读，脚本本身的代码甚至不会进入上下文，模型只看得到脚本的运行结果

这套机制解决的问题很直接：过去往上下文里塞的东西越多，模型能同时处理的"活的信息"就越少。渐进式披露让一个 Skill 理论上可以携带近乎无限的背景资料，因为大部分内容平时根本不会被加载。

最小的 SKILL.md 长这样：

---
name: my-first-skill
description: 这个 Skill 能做什么、什么场景该用它。这一行是触发的关键。
---

# 我的第一个 Skill

在这里写清楚具体的操作步骤、注意事项、输出格式要求。

三分钟体验

不写代码也能感受一次。目前主流的官方内置 Skill 覆盖了 Word / Excel / PPT / PDF 这类文档处理场景：你只需要正常提出需求，比如"帮我把这份数据整理成带格式的表格"，AI 会自己判断该不该调用对应的 Skill，全程不需要你了解背后的机制。

这一步的意义在于：先建立"啊原来这就是 Skill 在起作用"的直觉，再去看后面的原理和实操，会顺很多。

Skill 与其他概念的关系

这是最容易被单独一篇 Skill 科普文忽略的部分——因为 Skill 从来不是孤立存在的，它和 Agent、Context Engineering、Prompt Engineering、Harness Engineering、Loop Engineering 是同一套工程体系里互相咬合的几个齿轮。

  • × Agent　Skill 不给 Agent 增加智能，而是把一个通用的 Agent 变成某个具体领域的专家。Agent 是执行的载体，Skill 是装进这个载体里的专业知识包。
  • × Context Engineering　渐进式披露本质上就是一种上下文工程技巧——只在必要时才把信息放进上下文窗口。Skill 可以看作这套技巧的一个具体工程实现：用文件系统的目录结构，把"什么时候该读什么"变成了可执行的规则，而不是靠模型自己判断取舍。
  • × Prompt Engineering　SKILL.md 的正文说到底是一段会被"按需注入"的高阶提示词。可以说 Skill 把 Prompt Engineering 的成果从一次性的对话指令，变成了模块化、可复用、可版本管理的资产——这也是为什么很多人把 Skill 理解成"打包过的 Prompt"。
  • × Harness Engineering　SKILL.md frontmatter 里能声明允许调用哪些工具、加载哪些资源，三层加载机制本身也是一套运行时的调度规则。这些"AI 能在多大范围内自主行动"的边界设定，属于 Harness 层在管的事，Skill 依附在 Harness 之上运行，而不是替代它。
  • × Loop Engineering　好的 Skill 几乎不可能一次写完就用。真正成熟的做法是"写草稿 → 用真实 prompt 测试 → 评估触发率和输出质量 → 根据反馈修改"，反复几轮再上线。这个迭代闭环本身，就是 Loop Engineering 的一个具体案例。

对比：Skill / Prompt / MCP / RAG

开发者最容易搞混的四个概念，放在一张表里看会清楚很多：

Prompt / 记忆 MCP / 工具 RAG Skill
解决什么问题 单次或长期的行为指令 连接外部系统、执行具体动作 从外部知识库检索信息 沉淀"一件事该怎么做"的流程
关注点 说什么 能做什么 找什么 该怎么做
生命周期 对话内或长期偏好 随时可被调用 按查询实时检索 按需触发、分层加载
典型问题 每次都要重复交代 有能力但不知道该按什么流程组合使用 检索到了知识，但缺少"步骤感" 需要维护、迭代，写得不好会不触发

一句话总结分工：MCP 负责"能做什么"，Skill 负责"应该怎么做"，Agent 只是把两者串起来执行的载体。

十分钟上手：写自己的第一个 Skill

  1. 新建一个文件夹，比如 my-skill/
  2. 在里面放一个 SKILL.md，至少包含 name、description 两个字段，正文写清楚具体步骤
  3. 如果涉及需要稳定输出的操作（比如格式校验、数据排序），另建一个 scripts/ 目录放脚本，在正文里说明什么情况该调用它
  4. 如果参考资料比较多，另建一个 references/ 目录分文件存放，正文里只留"什么情况该翻哪个文件"的指路信息，而不是把所有细节都堆进 SKILL.md
  5. 用几个真实的提问句子测试一遍：它有没有在该触发的时候触发？输出是不是符合预期？
  6. 根据测试结果修改 description 和正文，再测一轮

到这一步，你已经走完了 Skill 从 0 到 1 的完整流程，剩下的都是细节打磨。

进阶经验

这部分是写给"有技术背景但还没真正落地过"的读者的——概念都懂，但动手写的时候容易在这几个地方栽跟头：

description 是触发的灵魂。 很多人把它写得过于简略，结果该触发的时候模型没反应——这是目前普遍存在的"欠触发"问题。经验是不仅要写清楚"这个 Skill 是干什么的"，还要主动列出具体的触发场景和关键词，宁可写得啰嗦一点，也不要惜字如金。

SKILL.md 别贪多。 正文建议控制在几百行以内，一旦感觉快超了，就该把细节拆到 references/ 目录里分层存放，SKILL.md 本身只保留主干流程和"该去哪个文件查"的指路牌。塞得太满，不仅挤占上下文，也会让模型抓不住重点。

能用脚本就别让模型现场生成代码。 对确定性要求高的任务——排序、格式校验、批量处理——预先写好脚本让模型直接调用执行，比每次现场生成一遍等效代码更省资源、更可靠，结果也更一致。

三个子目录职责要分清。 scripts/ 放可执行代码，references/ 放需要读进上下文的参考文档，assets/ 放模板、图标这类直接用于最终输出的静态文件。混着放，长期维护会很痛苦。

测试不是可选项。 写完不代表能用，一定要用几个贴近真实场景的提问跑一遍，看触发率和输出质量，再根据结果迭代。这一步做不到位，Skill 大概率会变成一个"写了但没人用得起来"的摆设。

常见踩坑

  • 欠触发：description 写得太笼统，导致该用的时候模型没主动去用，是目前最普遍的问题
  • 职责错位：把 Skill 当成连接外部系统的工具在用——那是 MCP 该干的事，Skill 里塞一堆 API 调用逻辑通常是设计思路走偏的信号
  • 安全性：社区已经出现过 Skill 被"投毒"的案例，尤其是那些会连接外部网络来源的 Skill，风险更高。建议只从可信来源安装，来源不明的一定要先审查内容再用——这本质上是一类新型的供应链安全问题，值得像对待"安装软件"一样谨慎对待
