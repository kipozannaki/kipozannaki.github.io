🔍 RAG Engineering 工业级落地

从"把文档塞进向量库"到"完整的知识供应链" · 双栖科普

跳转：引言 · 坐标 · 数据流 · Query 理解 · 检索 · 后处理 · 答案校验 · 进阶形态 · 实战避坑 · 结语

一、引言：为什么 RAG 不是"把文档塞进向量库"

很多人对 RAG 的理解停留在一个简单的流程图：用户问题 → 向量检索 → 注入 Prompt → 生成答案。结果上线后发现：检索回来的内容跟问题无关、上下文塞不下、模型还是瞎编。

问题出在哪？这三个地方都可能：

  • 问题本身就没问对——用户说"这个怎么用"，系统却去检索"使用场景介绍"，答非所问
  • 检索方案选错了——用专有名词查询却用纯向量检索，结果命中 90% 相似但版本完全不对的文档
  • 塞进去的方式不对——10 段检索结果全塞进去，不仅爆了 token 还让模型分不清哪个重要

RAG Engineering 干的事：从"问题理解"开始、经过"检索优化"、"结果后处理"、"答案校验"，设计一条从用户问题到模型答案的完整数据流。RAG 不是一个技术细节，而是一套系统工程。

💡 RAG 是"让 AI 在回答之前，先去翻一下资料"，但翻什么、怎么翻、翻回来怎么用，每一步都需要设计。

二、RAG 在工程体系里的坐标

这是关键的定位澄清：RAG 不是和 Context Engineering 平级，而是Context Engineering 的上游供应链。

Prompt Engineering Context Engineering RAG Engineering
管什么 单轮输入怎么写 单次会话里喂什么 窗口里那些东西从哪来
数据流 内向（用户输入） 流向（怎么组织） 来源（外部知识）
依赖关系 独立 依赖 Prompt 格式 为 Context 供给数据

换句话说：

  • 没有 RAG：Context 只能靠 Prompt 里的硬编码知识，外部信息无法实时获取
  • 有了 RAG：Context 可以动态地从知识库补充实时信息，模型回答的准确性和时效性都提升

RAG 和本站其他概念的关系：

  • × Prompt Engineering　RAG 检索结果最终要变成 Prompt 的一部分，所以检索结果的格式和表述方式要配合 Prompt 的结构
  • × Context Engineering　RAG 决定"这些知识从哪捞"，Context 决定"把它怎么摆进窗口"——两者协作
  • × Agent　Agent 可以自主决定"要不要查 RAG、什么时候查、查哪个知识库"，把 RAG 当成一个工具来用
  • × Loop Engineering　Loop 层可以定期刷新 RAG 的知识库，保证外部知识保持最新

三、从 Query 到 Answer：RAG 的完整数据流

用户问题
   │
   ▼
[1. Query 理解] ──→ 意图识别、问题改写、多跳拆解
   │
   ▼
[2. 检索] ──→ 向量检索、关键词检索、混合检索、重排序
   │
   ▼
[3. 后处理] ──→ 去重、截断、格式化、相关性过滤
   │
   ▼
[4. 注入 Context] ──→ 塞进 Prompt 的指定位置
   │
   ▼
[5. 生成] ──→ 模型基于"问题 + 检索结果"生成答案
   │
   ▼
[6. 答案校验] ──→ 幻觉检测、来源溯源、置信度评估

四、Query 理解：问题本身往往问不对

4.1 意图识别与澄清（Intent Clarification）

用户问题往往有歧义。比如"给我一份 React 教程"，可能指：

  • 从零开始学（想要基础教程）
  • 快速上手（想要进阶教程）
  • 查 API 文档（想要参考资料）

RAG 系统无法从单句话推断意图。处理方式：

  • 利用对话历史：如果这是多轮对话，前面的交互可能透露了用户的知识水平
  • 靠检索结果反推：检索 Top-5 结果差异很大，说明意图不清；如果很一致，说明问题表述清楚
  • 主动澄清：生成几个可能的意图，让用户选择（这在实战中很常见）
关键决策：是否为了 10% 的准确性提升而麻烦用户做一次选择？还是直接用最可能的意图进行检索？通常答案是"用 Top-3 意图的检索结果求并集"——兼顾覆盖面和成本。

4.2 问题改写（Query Rewriting）

用户的原始问题很少是"检索友好"的。

原始问题和改写后的对比：

原始问题 改写后 为什么要改
"这个怎么用？" "React useEffect 的依赖数组怎么填？" 前者太模糊，要靠对话历史补全
"为什么报错？" "TypeError: Cannot read properties of undefined 的常见原因和解决方案" 用具体错误信息替代模糊的"报错"
"对比一下 A 和 B" 拆解为两个独立查询 + 汇总指令 比较类问题需要分别检索再合并

技术方案：用 LLM 自动改写。Anthropic 的"HyDE"（Hypothetical Document Embedding）思路值得参考：让模型先写一个"理想答案"，然后用这个答案的 Embedding 去检索。

4.3 多跳拆解（Multi-hop Decomposition）

复杂问题需要拆成多个子查询，分别检索后汇总。

示例："2024 年 Q3 营收增长最快的 SaaS 公司，它的 CEO 之前在哪工作？"

查询 1: "2024 Q3 SaaS 公司营收增长率排名"
  结果: 公司 X 增长最快

查询 2: "公司 X CEO 履历和工作经历"
  结果: CEO 曾在 Y 公司任 CTO

关键：子查询之间存在依赖关系，不是并行检索能搞定的，需要 Workflow 或 Agent 层来编排。

五、检索：不只是向量相似度

5.1 向量检索（Dense Retrieval）

  • Embedding 模型选择：OpenAI text-embedding-3-large、BGE、E5——不是越大越好，要看领域匹配度。中文技术文档用 BGE-large-zh 往往比通用模型效果更好
  • Chunk 策略：按段落切、按固定 token 数切、按语义切——各有利弊。技术文档通常按标题层级切，chunk 大小 512，重叠 128
  • 陷阱：向量相似度 ≠ 语义相关。两个句子 Embedding 很近，但可能回答的是完全不同的问题

5.2 关键词检索（Sparse Retrieval）

BM25、TF-IDF 在这些场景下往往比向量检索更准：

  • 精确匹配（查询"Python 3.12 的 PEP 695"，BM25 能精确命中，向量可能把 3.11 版本也排前面）
  • 专有名词（人名、产品名、版本号）
  • 数字和代码片段

5.3 检索方案的决策框架

不同场景应该选什么检索方案？用这个决策树：

査询包含专有名词 / 数字 / 版本号？
├─ 是 → 混合检索（向量 + BM25）优于纯向量
└─ 否 → 纯向量检索通常够用

知识库大小？
├─ < 10K 文档 → 纯向量检索就行
├─ 10K-100K → 考虑混合检索
└─ > 100K → 混合 + 重排序（成本考虑）

能接受的推理延迟？
├─ 秒级响应 → 向量检索 + 轻量重排序
└─ 可容忍几秒 → 混合检索 + 重量级重排序

5.4 混合检索与重排序

混合检索：向量 + 关键词，结果融合。常见策略：

  • RRF（Reciprocal Rank Fusion）：按排名加权，不依赖绝对分数。推荐新手用这个，最稳健
  • 线性加权：dense_score * α + sparse_score * (1-α)，α 需调优

重排序：初筛召回 100 条，用更重的模型（cross-encoder）精排 top-10。权衡：慢、贵，但准确率提升显著。生产环境通常"轻量检索召回多 + 重排序精排少"。

六、后处理：检索回来不等于能用

6.1 去重与冗余消除

向量检索可能召回语义相近的多段内容，需要合并或去重。方法：聚类（如 HDBSCAN）+ 代表性片段选取。

6.2 截断与优先级

上下文窗口有限，检索回来 10 段内容可能只能塞 3 段。策略：

  • 按重排序分数截断（最简单）
  • 按信息类型分层（必含段 + 补充段）
  • 动态调整：简单问题少塞，复杂问题多塞

6.3 注入格式：因问题复杂度而异

简单问题的统一格式：

<documents>
[1] 来源: xxx.md | 置信度: 0.92
内容: ...

[2] 来源: yyy.md | 置信度: 0.88
内容: ...
</documents>

<instruction>
基于以上文档回答问题。如果文档中没有相关信息，明确说"我不知道"。
</instruction>

复杂问题的分层注入：

<primary_documents>  // 关键信息，必须有
...
</primary_documents>
<supporting_documents>  // 辅助信息
...
</supporting_documents>

多跳问题的链式注入：

<retrieved_documents step="1">
查询: "X 公司 2024 Q3 营收"
结果: ...
</retrieved_documents>
<retrieved_documents step="2">
查询: "X 公司 CEO 履历"
结果: ...
</retrieved_documents>

关键决策：问题的复杂度 → 选择相应的注入模式。每段内容都要标来源，方便后续溯源和幻觉检测。

七、答案校验：RAG 不是万能药

7.1 幻觉检测

模型可能"脑补"检索结果里没有的信息。检测方法：

  • 忠实度评分：用 NLI（自然语言推理）模型判断答案是否被检索内容支撑
  • 溯源检查：答案中的每个事实，能否追溯到某段检索内容的明确出处？

7.2 来源溯源

用户问"这个结论从哪来的"，系统需要能指出具体文档和段落。实现：检索阶段保留文档 ID + 段落偏移，生成阶段做 attribution 标记。

7.3 置信度评估与降级策略

检索结果质量和模型的"信心"往往不一样。即使检索结果充分，模型也可能"拒不承认"某个结论；反过来，模型也可能对一个只由单一来源支撑的结论过度自信。

置信度评估方法：

  • 自信度打分：让模型自己评估"我对这个答案有多确信"（0-1）
  • 检索覆盖度：回答中涉及 N 个事实点，其中有多少被检索结果覆盖？覆盖度 = 被支撑的事实数 / 总事实数
  • 来源多样性：是否来自单一文档还是多个独立来源？

根据置信度采取不同策略：

置信度 表现 策略
高（> 0.8） 检索结果充分支撑答案 直接回答
中（0.5-0.8） 检索结果部分相关，覆盖度 60-80% 回答 + 免责声明"基于有限信息"
低（< 0.5） 检索结果不相关或为空，覆盖度 < 60% 明确说"我不确定"，提议扩大检索范围

降级方案（当置信度低时）：

  • 扩大检索范围（Top-100 而不是 Top-10）
  • 切换 Embedding 模型（有时换一个模型效果差异很大）
  • 触发多跳检索（如果是多跳问题但当前只做了单跳）
  • 交给人工审核（关键决策类答案）

八、进阶：RAG 的三种演进形态

8.1 Naive RAG

单次检索 → 单次生成。最简单，也是大多数系统的现状。

8.2 Advanced RAG

在 Naive RAG 基础上加优化：

  • 查询改写、多跳拆解（Query 层）
  • 混合检索 + 重排序（检索层）
  • 后处理去重、分层注入（后处理层）
  • 置信度评估、降级策略（校验层）

这一篇讲的内容，基本就是 Advanced RAG 的完整体系。

8.3 Agentic RAG

Agent 自主决定"要不要检索、检索什么、检索几次、检索结果够不够用"。和 agent.html 讲的 Agent 概念直接对接：Agent 可以把 RAG 当成一个工具来用，也可以把 RAG 的"检索-评估-再检索"封装成自主决策循环。

九、实战避坑：真实项目中最常见的坑

坑一：文档更新后 Embedding 没同步，检索到的是过期信息

翻车现场：知识库一个月前的文档被索引了，现在文档改了，但索引没更新。用户问最新的信息，系统返回的是一个月前的回答。

根因：把 RAG 当成"一次索引、永久有效"的系统。

解决方案：

  • 给文档加版本和更新时间戳
  • 监听文档变更（webhook 或定期扫描），变更后自动增量更新索引
  • 定期清理过期文档（超过 X 天没更新的，或被标记为废弃的）
  • 这也是 Loop Engineering 要处理的问题：Loop 可以定期刷新 RAG 的知识库，但 RAG 层需要提供"哪些文档需要更新"的反馈机制
💡 RAG 不是"一次部署就完事"，需要持续的知识库维护。

坑二：Chunk 切太碎，跨段落的上下文丢失

翻车现场：一个技术概念分散在文档的不同段落，每个 chunk 都被独立索引和检索，模型拿到的是几个碎片化的片段，无法理解整体意思。

根因：机械地按 token 数截断。

解决方案：按文档结构切。技术文档按标题层级切，保留层级关系；通用文本可以按段落或语义单元切。

坑三：用户问题太模糊，检索结果散乱无法汇总

翻车现场：用户问"给我讲讲这个框架"，系统检索到了安装指南、API 参考、最佳实践三种完全不同的文档，模型试图把它们揉在一起，结果四不像。

根因：没有做意图澄清或问题改写。

解决方案：加一层问题理解。对模糊问题，要么让用户澄清、要么靠对话历史补全、要么用多个改写版本的检索结果求并集。

十、结语：RAG 是"知道得更准"的基础

给小白的话：RAG 就是"让 AI 在回答之前，先去翻一下资料"。但翻什么、怎么翻、翻回来怎么用，每一步都需要设计。好的 RAG 系统和坏的 RAG 系统的区别，不在"有没有向量库"，而在"问题理解得清不清、检索结果对不对、注入方式是否合理"这一系列细节上。

给开发者的话：RAG 系统的性能瓶颈很少在"向量检索有多快"，而在"检索回来的东西对不对、够不够、模型有没有用上"。优化 RAG 要从数据流的全链路看——从 Query 理解开始、经过检索方案选择、后处理优化、答案校验——每一层都是优化空间。光调 Embedding 模型是没用的。

系列衔接：RAG 把外部知识接进了 Context，Context Engineering 决定这些知识在窗口里怎么摆，Prompt Engineering 决定怎么告诉模型"基于这些知识回答"。三层协作，才能让模型真正用上外部知识。而 Loop Engineering 在背后确保知识库保持最新，Agent 在上层自主决定要不要用 RAG、什么时候用。

💡 RAG 不是万能药，但它是现代 AI 应用的标配。一旦系统涉及外部知识，RAG 层的设计质量往往决定了整体系统的天花板。
