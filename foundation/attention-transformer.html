🧠 Attention：一篇论文凭什么把标题写成「你只需要注意力」？

为什么上下文越长推理越贵、为什么模型会「中间遗忘」？一篇讲清 Transformer 的发动机 · 双栖科普

跳转：引言 · 坐标 · 为什么不能直接用 RNN · 怎么工作的 · 怎么影响能力 · 为什么开发者要懂 · 常见误区 · 总结

零、引言：一篇论文，把整个行业的路线图给改了

2017 年，Google 一篇论文的标题只有五个词：Attention Is All You Need——「你只需要注意力」。八年后回头看，这篇论文里提出的 Transformer 架构，撑起了从 GPT 到几乎所有现代大模型的底座。

先扔几个问题：为什么上下文一变长，推理速度和成本会明显往上跳？为什么关键信息塞进 Prompt 中间位置，模型经常「看不见」？为什么模型训练时见过的最长上下文是 4K，你硬塞 32K 进去，它开始「没那么聪明」了？

💡 Embedding 让每个词有了一个初始位置。Attention 要做的，是让这个位置随着上下文实时「挪动」。

一、Attention / Transformer 在 AI 世界里的坐标

Tokenizer → Embedding → Attention / Transformer（在向量之间动态传递信息）← 本篇 → Pretraining → Inference → LLM 输出
工程概念跟 Attention / Transformer 的关系
Context Engineering上下文为什么「越长质量越容易滑坡」，根子就在注意力分布规律
RAG Engineering检索出来的多个 chunk 摆放顺序会影响注意力权重
推理优化Attention 的 O(n²) 计算复杂度直接决定了推理要花多少显存
Long Context本篇讲清 Attention 本身的机制和局限

二、为什么不能直接用 RNN 一步步处理？

方案致命问题
RNN / LSTM逐 Token 顺序处理，没法并行训练；序列一长，早期信息在层层传递中被稀释
加大隐藏状态维度治标不治本，长距离依赖问题依然存在
CNN 卷积处理序列单层只能看到窗口内的局部关系

Attention 的思路：不再依赖「一步步传递」的记忆，而是让每一个 Token 直接和序列里所有其他 Token 计算关联度，一步到位。

💡 RNN 是「传纸条」。Attention 是「开一次全员会议」——每个 Token 都能直接和所有其他 Token 对话。

三、Attention 到底是怎么工作的？

每个 Token 会算出三个向量：Query（我在找什么）、Key（我这里有什么）、Value（我实际携带的内容）。拿 Query 去和所有 Key 做匹配，匹配度高的文档，它的 Value 就会在最终结果里占更大权重。

两个关键设计：多头注意力（Multi-Head Attention）——并行算好几组不同的 Query/Key/Value，让不同的「头」分别学会关注不同类型的关系；位置编码（Positional Encoding）——Attention 本身对词的顺序不敏感，必须额外注入位置信号。

四、Attention 会如何影响模型能力？

为什么上下文越长，推理越贵、越慢？

Self-Attention 计算量随序列长度呈平方级增长——上下文长度翻倍，计算量大约变成四倍。

为什么关键信息放在长 Prompt 中间，模型经常「看不见」？

这就是「Lost in the Middle」（中间遗忘）现象——实际训练出来的模型普遍对序列开头和结尾更敏感，中间部分的信息容易被「稀释」。

为什么训练时 4K 上下文的模型，硬塞 32K 进去会「变笨」？

位置编码是在训练阶段针对特定长度范围学出来的，超出范围就是外推（extrapolation）失败。

五、为什么开发者一定要懂 Attention？

「Why」：Context Engineering 里「精简上下文」的原则，根源在 O(n²) 计算成本和「中间遗忘」这两个底层约束。RAG 里检索结果的排序策略，「中间遗忘」意味着最相关的 chunk 放在开头或结尾效果更好。

七、总结：一场没有中间人的全员会议

给小白的话：下次听到「这个模型上下文支持 32K」，可以多问一句——支持是技术参数，好不好用是另一回事。

给开发者的话：理解 Self-Attention 的 O(n²) 代价、理解「中间遗忘」是经验规律而非偶发 bug——这两件事打通了，Context Engineering 里「为什么要精简上下文」、RAG Engineering 里「为什么排序要讲究」会一下子全部串起来。

💡 Embedding 给了每个词一个起点，Attention 让这个起点学会了看着场合变。
