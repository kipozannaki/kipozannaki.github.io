⚡ Inference：为什么第一个字慢，后面的字却快？

为什么长对话越聊越卡越贵？为什么同一个模型不同服务商速度差好几倍？一篇讲清推理到底在算什么 · 双栖科普

零、引言：模型说话，其实是「边想边说」

先扔几个问题：

  • 为什么调用 API 的时候，第一个字往往要等一下，后面的字却几乎是秒吐出来的？
  • 为什么同一段对话，聊到第五十轮的时候，明显比第一轮慢，也更贵？
  • 为什么同一个模型权重，换一家服务商跑，速度和价格能差出好几倍？
  • 为什么显存不够会直接导致模型跑不起来，而不是「算得慢一点」这么简单？

这些现象的答案，都在 Inference（推理）这一步——也就是模型对齐训练完成之后，被真正拿去部署、每次你调用 API 时实际发生的事情。上一篇讲的 Alignment 让模型学会了「该怎么回应」，本篇要讲的，是它把答案一个字一个字「说出来」的这个过程，到底在算什么。

💡 训练阶段模型是在「学」，推理阶段模型是在「用」——你每天调用 API 感受到的延迟和成本，全部发生在推理这一步。

一、Inference 在 AI 世界里的坐标

紧接 Alignment 管线往下走：

Tokenizer → Embedding → Attention / Transformer → Pretraining → Alignment（对话模型训练完成）
    ↓
Inference：Prefill（读懂输入）→ Decode（逐字生成）  ← 本篇
    ↓
LLM 输出

这一阶段的影响会渗透到整个工程体系里：

工程概念跟 Inference 的关系

💡 Attention 那篇讲过计算量随上下文长度呈平方级增长——这不是一个抽象的数学结论，你每天感受到的「聊得越久越卡」，就是这条规律在真实世界里的样子。

二、为什么不能一次性把整句话生成出来？

一个直觉的问题：模型训练时可以并行处理整个句子，为什么推理时不能一次性把答案整段吐出来？

方案怎么做致命问题
一次性并行生成整段输出所有位置的词同时预测出来每个词在被预测的那一刻，还看不到后面即将出现的词是什么，很难保证整段话前后连贯、语法一致，实际效果明显不如逐步生成
逐词生成（自回归，Autoregressive）每次只生成一个新词，把它加进上下文，再生成下一个生成一段长回答必须严格串行，没法把单条请求内部的生成过程并行加速

模型最终采用的是自回归生成：每一步只能根据「已经生成出来的内容」去预测下一个词，没法跳过，也没法回头。这是保证输出连贯、逻辑一致的代价——训练阶段能并行处理整个序列，是因为训练时「正确答案」已经全部摆在那里；推理阶段要生成的内容还不存在，只能一步一步「边想边说」。

💡 训练是「照着答案抄一遍」，可以整页一起抄；推理是「现场即兴发言」，只能一句接一句往下说，说出口的话没法收回去重排。

三、Inference 到底是怎么工作的？

推理过程分两个阶段，性质完全不同：

Prefill（预填充）：处理你输入的全部内容——问题、历史对话、检索回来的文档，一次性并行计算它们之间的 Attention。因为输入内容是已知、完整的，这一步可以并行处理，但如果输入本身很长（比如塞了一大段检索文档），这一步照样要花不少时间——这正是「第一个字要等一下」的原因：首字延迟（TTFT，Time to First Token）主要取决于 Prefill 阶段要处理多长的输入。

Decode（解码）：生成阶段，每次只产出一个新 Token，加进上下文后再生成下一个——这一步严格串行，没法跳过。每生成一个新字，理论上都要把「当前已生成的全部内容」重新和历史做一次 Attention 计算，但生产系统通常会用 KV Cache（把已经算过的 Key/Value 缓存下来，新 Token 只需要计算自己，复用之前的结果）来避免重复计算——这大大提了速，但代价是显存占用会随着对话长度不断增长。

这也是为什么显存常常比算力更早成为瓶颈：模型参数本身要占显存，KV Cache 也要占显存，而且 KV Cache 的占用量是随「对话长度 × 并发请求数」一起涨的——一台服务器能同时撑住多少用户，很多时候不是被算力卡住，是被显存卡住的。KV Cache 具体怎么被工程化管理（比如 PagedAttention 这类技术），留给 Topic③ 专题细讲。

💡 Prefill 是「读完你说的话」，Decode 是「一个字一个字回答你」——前者可以并行冲刺，后者只能老老实实排队，这是两种完全不同的性能特征。

四、Inference 会如何影响模型能力？

只讲真实案例。

为什么第一个字要等好几秒，后面的字却秒回？

翻车现场：调用 API 时，明显感觉到发出请求后有个短暂的停顿，等第一个字出来之后，剩下的内容却几乎是流畅吐出来的。

根因：这正是 Prefill 和 Decode 两个阶段性质不同导致的——Prefill 要先把整个输入「读完想清楚」，输入越长（比如带了很长的历史对话或检索文档），这一步耗时越长；Decode 阶段是在已经「理解完」输入的基础上逐字吐出结果，配合 KV Cache 复用，单字生成速度通常快得多。这也是为什么生产系统里，首字延迟（TTFT）和每字生成速度（TPOT）要分开监控和优化，而不是笼统看一个「响应时间」。

为什么长对话越聊到后面，响应越来越慢、越来越贵？

翻车现场：同一个对话窗口一直聊下去，聊到几十轮之后，感觉每次回复都比一开始慢，账单涨得也比预期快。

根因：每生成一个新字，都要对「当前全部历史」重新计算一次 Attention——即使有 KV Cache 优化，计算量依然会随着历史长度增长而增长，同时 KV Cache 占用的显存也在跟着涨。这也是为什么很多产品在对话轮数变多之后，会建议「新开一个对话」——本质是在绕开这个物理约束，不只是为了界面简洁。

为什么同一个模型，不同服务商跑出来的速度和价格差异巨大？

翻车现场：调用同一款开源模型，换一家托管服务，响应速度和计费单价能差出好几倍。

根因：推理效率高度依赖工程实现——批处理调度做得好不好（比如能不能把多个用户的请求高效拼在一起处理）、显存管理方案先不先进、有没有用推测解码等加速手段，这些都是纯工程优化空间。同样的模型权重，跑在不同的推理引擎上，性能可以差出几倍——这部分具体技术，会放到 Topic③ 专题细讲。

💡 模型权重决定了「能不能答对」，推理引擎决定了「答得快不快、贵不贵」——两者是完全不同的两件事，很多时候慢不是模型的锅。

五、为什么开发者一定要懂 Inference？

Context Engineering：长对话越聊越卡、越贵的根源，就在本篇讲的 KV Cache 随长度增长的机制——理解了这一点，「精简上下文」「定期开新对话」这些应用层原则，才不只是经验之谈，而是有物理约束支撑的判断。

Topic③ 推理优化：本篇讲清楚了「为什么慢、为什么占显存」这些原理性问题，vLLM、SGLang、PagedAttention、Speculative Decoding 这些具体工程手段，都是在针对这些瓶颈做优化——理解了原理，才能真正看懂这些工具具体在解决什么问题，避免只会照抄配置参数。

Harness Engineering：设计 Agent Loop 的超时和重试策略时，首字延迟和逐字生成速度是两个独立的性能指标，用同一个笼统的超时值去覆盖两种完全不同的等待场景，很容易踩坑。

往下一篇的伏笔：推理这一层讲的是「跑一个标准模型该怎么算」，但不是所有模型的参数都会被同时用上——下一篇 MoE（混合专家模型）要讲的，正是「参数量」这个数字为什么开始变得没那么诚实。

六、常见误区

误区一：模型推理慢，主要是因为算力不够

真相：很多情况下真正的瓶颈是显存带宽和显存容量，而不是纯算力——尤其是 Decode 阶段，是「访存密集型」而不是「计算密集型」的过程，单纯换更强的算力芯片不一定能显著提速，需要针对性做显存管理优化。

误区二：上下文变长，只是多花一点点计算量而已

真相：Attention 的计算量随上下文长度呈平方级增长，KV Cache 的显存占用则随长度线性增长——长对话对速度和成本的影响是叠加的，不是线性的小幅增加。

误区三：推理速度只取决于模型本身，跟部署方式没关系

真相：同样的模型权重，配合不同的推理引擎、批处理策略、量化方案，实际吞吐量和延迟可以有数倍差异——这是一整个独立的工程领域，会在 Topic③ 细讲。

误区四：首字延迟和每字生成速度是同一件事，统称「推理速度」就够了

真相：这是来自 Prefill 和 Decode 两个不同瓶颈的独立指标，优化其中一个不代表另一个也会变好，生产系统需要分别监控、分别优化。

💡 理解推理，本质是理解你每天付的 API 账单和等待的那几秒钟，到底是在为哪一步买单——是在「读你说的话」，还是在「一个字一个字回答你」。

七、总结：说话是即兴的，没法回头重排

给小白的话：下次觉得模型「怎么突然卡了一下才开始说话」，可以想想——它不是在犹豫，是在认真读完你说的所有话（包括历史对话），才开口的。

给开发者的话：Inference 是训练成果第一次真正被「消费」的地方——Prefill 和 Decode 两个阶段性质完全不同、KV Cache 用显存换速度、显存常常比算力更早成为瓶颈，这三件事打通了，才能看懂为什么长对话会变贵变慢，也才能真正理解 Topic③ 那些推理优化工程手段究竟在解决什么问题，而不只是照着文档调参数。

💡 训练决定了模型知道什么、愿意怎么说，推理决定了这些话说出来要花多久、花多少钱——后者往往才是生产环境里最先让人头疼的那一半。
